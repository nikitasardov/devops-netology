## Постмортем инцидента GitHub от 21 октября 2018 года

| | |
|---|---|
| **Краткое описание** | 21 октября 2018 года в 22:52 UTC замена неисправного сетевого оборудования привела к потере связи с основным дата-центром на восточном побережье США на 43 секунды. Это вызвало цепную реакцию: автоматическое переключение MySQL-кластеров на западное побережье, рассинхронизацию данных между дата-центрами и деградацию сервиса на 24 часа 11 минут. |
| **Причина инцидента** | Кратковременное разделение сетей дата-центров привело к тому, что система управления MySQL-топологией по Raft-консенсусу выбрала новые primary-ноды в дата-центре западного побережья. После восстановления связи оба дата-центра содержали уникальные записи, отсутствующие в другом, — обратное переключение стало невозможно без потери данных. |
| **Воздействие** | Деградация сервиса GitHub.com в течение 24 часов 11 минут. Пользователи видели устаревшие и несогласованные данные (issues, pull requests, комментарии). Вебхуки и сборка GitHub Pages были приостановлены. Более 5 млн webhook-событий и 80 тысяч Pages-сборок скопились в очереди. ~200 000 webhook-событий были потеряны из-за истечения внутреннего TTL. |
| **Обнаружение** | В 22:54 UTC (через 2 минуты) внутренний мониторинг начал генерировать алерты. К 23:02 UTC первая команда реагирования определила, что топологии множества MySQL-кластеров находятся в неожиданном состоянии. |
| **Реакция** | 23:07 — заблокирован деплой. 23:09 — статус переведён в yellow. 23:11 — подключён координатор инцидента. 23:13 — статус переведён в red, вызвана команда database engineering. Принято решение приоритизировать целостность данных над скоростью восстановления. |
| **Восстановление** | К 00:41 UTC 22 октября запущено восстановление всех затронутых MySQL-кластеров из бэкапов (хранились удалённо в облаке), что заняло часы из-за объёмов (терабайты). К 11:12 UTC primary-ноды восстановлены на восточном побережье, но реплики отставали — пользователи видели несогласованные данные. К 16:24 — реплики синхронизированы и переключены на исходную топологию. К 23:03 UTC — обработаны все очереди, статус green. |
| **Таймлайн** | 22:52 — потеря связи между дата-центрами на 43 сек, автоматическое изменение топологии БД в западном дата-центре |
