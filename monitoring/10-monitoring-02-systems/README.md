5. Опишите основные плюсы и минусы pull и push систем мониторинга.

```
ОТВЕТ:
- push-модель, плюсы:
  - проще отправлять данные в несколько систем мониторинга
  - гибкая настройка - агент решает, когда отправлять данные
  - используя UDP (менее затратный протокол) можно повысить производительность сбора метрик
- push-модель, минусы:
  - агент сам отправляет данные, серверу сложнее контролировать подлинность
  - при множестве агентов сложнее обеспечить безопасность
  - сложнее отладка, если данные не приходят, причины могут быть разнообразные, т.к. много агентов и у каждого могут быть свои особенности
- pull-модель, плюсы:
  - легче контролировать подлинность данных, когда сервер сам опрашивает агентов
  - можно настроить единый прокси-сервер с шифрованием для всех агентов
  - легче отладка, т.к. сервер единообразно запрашивает данные и проще понять причины неполадок
- pull-модель, минусы:
  - сложнее отправлять данные в несколько систем, т.к. сервер инициирует сбор данных
  - менее гибкая настройка отправки, т.к. сервер контролирует частоту сбора данных
  - обычно используется TCP, что может быть более затратно в сравнении с UDP
```

| Аспект | Push-модель | Pull-модель |
|--------|-------------|-------------|
| **Репликация данных** | ✅ Проще отправлять данные в несколько систем мониторинга или резервные копии | ❌ Сложнее, т.к. сервер инициирует сбор данных |
| **Гибкость настройки** | ✅ Агент решает, когда отправлять данные | ❌ Сервер контролирует частоту сбора данных |
| **Производительность** | ✅ UDP менее затратен, выше производительность | ❌ Обычно TCP, более затратно |
| **Контроль подлинности** | ❌ Сложнее контролировать подлинность данных | ✅ Легче контролировать, когда сервер сам опрашивает |
| **Безопасность** | ❌ При множестве агентов сложнее обеспечить безопасность | ✅ Можно настроить единый прокси-сервер с TLS |
| **Отладка** | ❌ Сложнее отладка, причины могут быть разнообразные | ✅ Легче отладка, сервер единообразно запрашивает данные |

9. Изучите список [telegraf inputs](https://github.com/influxdata/telegraf/tree/master/plugins/inputs). 
Добавьте в конфигурацию telegraf следующий плагин - [docker](https://github.com/influxdata/telegraf/tree/master/plugins/inputs/docker):
```
[[inputs.docker]]
  endpoint = "unix:///var/run/docker.sock"
```

Дополнительно вам может потребоваться донастройка контейнера telegraf в `docker-compose.yml` дополнительного volume и 
режима privileged:
```
  telegraf:
    image: telegraf:1.4.0
    privileged: true
    volumes:
      - ./etc/telegraf.conf:/etc/telegraf/telegraf.conf:Z
      - /var/run/docker.sock:/var/run/docker.sock:Z
    links:
      - influxdb
    ports:
      - "8092:8092/udp"
      - "8094:8094"
      - "8125:8125/udp"
```

После настройке перезапустите telegraf, обновите веб интерфейс и приведите скриншотом список `measurments` в веб-интерфейсе базы telegraf.autogen . Там должны появиться метрики, связанные с docker.
